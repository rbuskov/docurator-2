## Spec 002 — connect-gmail-accounts

### F-004 — OAuth scope policy contradicts architecture.md, vision.md, and itself

> **Status: FIXED.** vision.md and architecture.md updated to acknowledge the three-scope set (`gmail.readonly` + `openid` + `userinfo.email`) with explicit framing that only `gmail.readonly` touches Gmail; the identity scopes grant only the user's email address for the OAuth handshake. Spec 002's acceptance criteria and "Other" deliverable updated to match. Spec 005's acceptance criterion updated. Spec 003 and Spec 006 acceptance criteria were already aligned with the three-scope set. Merged-in F-014 also resolved by these edits.

- **Type:** Inconsistency
- **Severity:** High
- **Specs:** 002, 005
- **Section:** Implementation notes vs Acceptance criteria; spec vs architecture.md vs vision.md
- **Description:** `architecture.md` § "Read-only Gmail access" and `vision.md` § "Read-only access to email" both commit to the app requesting **only** `gmail.readonly`. Spec 002's Implementation notes deliberately add `openid` and `https://www.googleapis.com/auth/userinfo.email` to the consent request. Spec 002's own Acceptance criteria contradict that plan ("opens consent screen for `gmail.readonly` (and only `gmail.readonly`)" and "no references to any non-`gmail.readonly` OAuth scope"). Spec 003's acceptance criterion subsequently allows `gmail.readonly`, `userinfo.email`, and `openid`. Spec 005 reverts to "zero references to non-`gmail.readonly` OAuth scopes" — disagreeing with both Spec 002 (Implementation) and Spec 003 (Acceptance). The inconsistency cascades through three specs and contradicts both architecture.md and vision.md.
- **Evidence:**
  - architecture.md § Read-only Gmail access: "Per account, the app requests **only** `https://www.googleapis.com/auth/gmail.readonly`. It does not request ... any other scope."
  - vision.md § Read-only access to email: "Docurator requests only `gmail.readonly` from Google, for every account the user connects."
  - Spec 002 Implementation notes: "Include `openid` and `https://www.googleapis.com/auth/userinfo.email` in the consent request alongside `gmail.readonly`."
  - Spec 002 Acceptance criteria: "consent screen for `gmail.readonly` (and only `gmail.readonly`)" / "no references to any non-`gmail.readonly` OAuth scope"
  - Spec 003 Acceptance criteria: "zero references to OAuth scopes other than `gmail.readonly`, `userinfo.email`, and `openid`"
  - Spec 005 Acceptance criteria: "zero references to non-`gmail.readonly` OAuth scopes"
- **Suggested fix:** Pick one resolution: either (a) require `userinfo.email`/`openid` (and update vision.md + architecture.md to acknowledge the additional read-only scopes), or (b) drop the extra scopes and read the email some other way (e.g. an HTTP `userinfo` call after token exchange, but that requires a scope too). All three specs and both governing docs need to land on the same answer.
- **Merged from:** F-014
