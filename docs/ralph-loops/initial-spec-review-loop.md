# Spec Review Loop

You audit the implementation specs for the Docurator project, looking for inconsistencies, gaps, and overlaps. This prompt runs in a loop. Each iteration you do **one** focused unit of review work and stop. The next iteration picks up where you left off by reading the filesystem.

The goal of the loop: produce a thorough, organized review of every spec under `specs/` and the relationships between them, written to `initial-spec-review-findings.md`. The review is **read-only** with respect to the specs and the surrounding documents — you find issues and document them; you do not fix them.

---

## Inputs to read every iteration

- `VISION.md` — what Docurator is and why it exists. Reference for "is this spec consistent with the project's purpose?"
- `ARCHITECTURE.md` — how Docurator is built. Reference for "is this spec consistent with the technical design?"
- `SLICES.md` — the ordered list of vertical slices. Each slice should have a corresponding spec.
- `specs/000-template.md` — the structural template every spec should follow.
- `specs/NNN-*.md` — the specs being reviewed.
- `initial-spec-review-findings.md` — your in-progress review. Read this first every iteration to find out where you left off.

You don't need to re-read every section in detail every iteration. Skim what you've worked with before. Re-read closely when investigating a specific finding.

## Outputs you write

- `initial-spec-review-findings.md` — the only file you create or modify. Every iteration appends findings to it or ticks off a checklist item in it.

You do **not** modify `VISION.md`, `ARCHITECTURE.md`, `SLICES.md`, `specs/000-template.md`, or any `specs/NNN-*.md` file. The point of this loop is to produce a review report; fixes happen elsewhere.

---

## The findings file structure

`initial-spec-review-findings.md` is both the output and the loop's progress tracker. Its top section is a checklist. The loop is done when every checkbox is ticked.

When the file does not yet exist, the first iteration creates it with this structure:

````markdown
# Docurator — Spec Review Findings

Generated by the spec review loop. This document is the audit report for the specs in `specs/`. It is built up incrementally — each unchecked checkbox below is one unit of review work.

## Review progress

### Per-spec review

- [ ] 001-<name>
- [ ] 002-<name>
- ...
- [ ] NNN-<name>

### Cross-cutting checks

- [ ] CC1 — Prerequisite/Deliverable resolution matrix
- [ ] CC2 — Identifier vocabulary consistency
- [ ] CC3 — Orphaned Deliverables (produced but never consumed)
- [ ] CC4 — SLICES.md alignment with specs/
- [ ] CC5 — Out-of-scope forward references
- [ ] CC6 — Acceptance criteria coverage
- [ ] CC7 — VISION.md alignment
- [ ] CC8 — ARCHITECTURE.md alignment
- [ ] CC9 — Template conformance

### Consolidation

- [ ] CO1 — Dedupe and merge overlapping findings
- [ ] CO2 — Severity-sorted summary

## Findings

(Sections are appended below as the review progresses.)
````

Each finding gets a structured entry under the appropriate section heading. Finding format:

````markdown
### F-NNN — <short title>

- **Type:** Gap | Overlap | Inconsistency | Minor
- **Severity:** High | Medium | Low
- **Specs:** 003, 005
- **Section:** Prerequisites | Deliverables | Detailed design | ...
- **Description:** What's wrong, in concrete terms.
- **Evidence:** Direct quote or specific reference (file + section).
- **Suggested fix:** If obvious. Otherwise leave blank.
````

Findings are numbered globally (F-001, F-002, …) in the order they're discovered, so any spec or reviewer can reference a finding by its ID.

---

## The single decision you make each iteration

Take **exactly one** of the following actions, in this priority order. Stop after you take one action.

1. **No findings file?** Create `initial-spec-review-findings.md` using the structure above. Populate the per-spec checklist by listing every `specs/NNN-*.md` file (excluding `000-template.md`). Stop.

2. **Per-spec checklist has unchecked items?** Pick the lowest-numbered unchecked spec. Read it carefully alongside `000-template.md`, `VISION.md`, and `ARCHITECTURE.md`. Run every per-spec check listed below. Append a `## Spec NNN — <name>` heading under Findings, then list each issue as a Finding entry. If you find nothing, write "No findings." under the heading. Tick the checkbox. Stop.

3. **Cross-cutting checklist has unchecked items?** Pick the first unchecked cross-cutting check. Run it across every spec. Append a `## Cross-cutting — CCN: <name>` heading under Findings, then list each issue. If you find nothing, write "No findings." Tick the checkbox. Stop.

4. **Consolidation checklist has unchecked items?** Pick the first unchecked consolidation step. Do it. Tick the checkbox. Stop.

5. **Everything checked?** Output `LOOP DONE` and stop. Make no changes.

If two conditions look true at once, the higher-priority one wins.

---

## Per-spec checks

For each spec, run all of the checks below. Report every issue as a separate Finding.

### Structure

- Does the spec match `000-template.md`? All sections present, in the right order, with the right headings?
- Is `Status` present and either `draft` or `ready`?
- Is `Observable result` a single sentence describing user-visible behavior?

### Prerequisites (Consumes)

- Does every category in the fixed vocabulary appear (with `—` where empty), or has a category been silently dropped?
- Does every listed item resolve **verbatim** to a Deliverable in some earlier spec? Same string, same casing, same level of qualification (e.g. `documents.amount_edited`, not `amount_edited` or "the amount-edited flag").
- Is anything listed that's actually being introduced by this spec (i.e., should be a Deliverable, not a Prerequisite)?

### Deliverables (Produces)

- Does every category appear (with `—` where empty)?
- Is each item concrete? "An endpoint to trigger sync" is a finding; "`POST /api/sync` accepting `{ since?: ISO8601 }` and returning a job id over SSE" is acceptable.
- Is each item produced **only** by this spec? If two specs both list the same Deliverable, that's an overlap.
- Is the item phrased the same way it's later consumed? Mismatched names cause silent gaps.

### Out of scope

- Does each item reference a specific later slice ("→ Slice NNN")?
- Does the referenced slice exist?
- Is the slice numbered higher than the current spec (forward-only references)?

### Detailed design

- Does it cover each Deliverable in the list?
- Does it reference `ARCHITECTURE.md` sections by name rather than restating them?
- Does it introduce architecture not present in `ARCHITECTURE.md`? If so, is that introduction flagged in Risks / open questions?

### Acceptance criteria

- Are they testable? "It works correctly" is not; "Clicking Sync produces a `processed_messages` row for every fetched message" is.
- Does at least one criterion exercise the path described in `Observable result`?
- Are they observable behavior, not implementation details?

### Risks / open questions

- If `ARCHITECTURE.md` is silent on something the spec had to decide, is the decision flagged here?
- Are there obvious unknowns the spec quietly assumes its way past?

## Cross-cutting checks

These run across multiple specs. Each is its own iteration.

### CC1 — Prerequisite/Deliverable resolution matrix

For every Prerequisite in every spec, find which earlier spec produces it (verbatim). Build a mental table:

```
Spec | Prerequisite               | Produced by
003  | documents.amount_edited    | (NONE)        ← gap
005  | sync_state                 | 004           ← OK
...
```

Report each unresolved Prerequisite as a Gap. Cite both the consuming spec's section and the search you did to confirm no earlier spec produces it.

### CC2 — Identifier vocabulary consistency

Pick out identifiers that cross spec boundaries: table names, column names, API endpoint paths and methods, component names, env var names, file paths, status enum values, model names. For each, check it's spelled and cased identically everywhere it appears. Common drift:

- `processed_messages` vs `processedMessages`
- `POST /api/sync` vs `POST /sync`
- `review_status = 'approved'` vs `review_status = 'Approved'`

Report each drift as an Inconsistency.

### CC3 — Orphaned Deliverables

For every Deliverable across all specs, check whether a later spec lists it as a Prerequisite, or whether it's user-visible (mentioned in Observable result or Acceptance criteria).

A Deliverable that's neither consumed nor user-visible might be unnecessary — flag it as Minor unless it's clearly internal scaffolding (e.g., a migration runner used implicitly by every later migration).

### CC4 — SLICES.md alignment with specs/

Compare `SLICES.md` and the `specs/` directory:

- Same count of items (excluding `000-template.md`)?
- Same order?
- Same kebab-case names?
- For each slice, does the bullet description in `SLICES.md` agree with the spec's `Observable result`? (Brevity is fine; contradiction is not.)

Report mismatches.

### CC5 — Out-of-scope forward references

Across all specs, collect every "→ Slice NNN" reference. For each:

- Does the referenced slice exist?
- Is it numbered higher than the spec doing the referencing?
- Does the referenced spec's Deliverables actually cover the deferred work?

Report broken references and "deferred to Slice NNN, but Slice NNN doesn't actually do it" cases.

### CC6 — Acceptance criteria coverage

For each spec, verify that at least one Acceptance criterion exercises the `Observable result`. A spec whose Observable result is "I can click Sync and watch progress" should have a criterion that involves clicking Sync and observing progress.

Report mismatches as Inconsistencies.

### CC7 — VISION.md alignment

Read each spec against the principles in `VISION.md`. Look especially for:

- Privacy violations (anything sending email content off the machine)
- Gmail write operations (the read-only commitment)
- "Hoarding" (storing email content beyond confirmed receipts)
- Lock-in (formats or behaviors that would trap user data)
- Anything contradicting the "core principles" section

Any spec that quietly contradicts a vision principle is a High-severity Inconsistency.

### CC8 — ARCHITECTURE.md alignment

Read each spec against `ARCHITECTURE.md`. Look for:

- Components named or described differently from `ARCHITECTURE.md`
- Tech stack deviations not flagged in Risks
- Storage decisions that differ from the schema in `ARCHITECTURE.md`
- Flow descriptions that contradict the Key flows section

Any deviation that isn't flagged in the spec's Risks/open questions is an Inconsistency.

### CC9 — Template conformance

For each spec, diff its structure against `000-template.md`:

- Same section headings, in the same order?
- All required categories present in Prerequisites and Deliverables (using `—` for empty)?
- No extra top-level sections invented?

Report each deviation, even small ones, as Minor.

## Consolidation

After all per-spec and cross-cutting checks are done, the findings file may have duplicate or overlapping issues recorded in different sections. Two final iterations clean it up.

### CO1 — Dedupe and merge overlapping findings

Walk the Findings section. Where two or more findings describe the same underlying issue (e.g., a missing Deliverable surfaced once during the per-spec review of the consumer and again in CC1), merge them into a single canonical finding. Keep the lowest finding ID; cross-reference the merged-in IDs in a "Merged from: F-XXX, F-YYY" line on the survivor.

### CO2 — Severity-sorted summary

Insert a Summary section just below the Review progress section:

````markdown
## Summary

- **High severity:** N findings — F-001, F-007, F-012, …
- **Medium severity:** M findings — …
- **Low severity:** K findings — …
- **By type:** X gaps, Y overlaps, Z inconsistencies, W minor.

Top issues to address first:
1. F-001 — <short title>
2. F-007 — <short title>
3. …
````

Pick the top issues by severity first, then by how many specs they affect.

## Quality bar for findings

- **Be specific.** "Spec 005 has a problem" is not a finding. "Spec 005 lists `documents.amount` in Prerequisites, but no earlier spec produces this column — closest match is in 008" is.
- **Cite evidence.** Quote the offending text or reference the section. Reviewers should be able to verify a finding without re-reading both specs.
- **Don't propose architecture.** If a fix is non-obvious, leave Suggested fix blank rather than inventing a design. The review surfaces problems; deciding the fix is a separate task.
- **Don't restate the spec.** Findings are deviations from the spec's own contract, not summaries of the spec.
- **One issue per finding.** If you find three problems with one spec, that's three findings, not one.
- **Calibrate severity.**
  - **High** — gaps that block the slice from being implementable, or contradictions with `VISION.md` / `ARCHITECTURE.md`.
  - **Medium** — overlaps, inconsistent vocabulary across specs, untestable acceptance criteria, broken forward references.
  - **Low / Minor** — formatting, missing `—` placeholders, prose that duplicates `ARCHITECTURE.md` instead of referencing it.

---

## What "done" means

The loop is finished when **all** of these hold:

- `initial-spec-review-findings.md` exists.
- Every checkbox in Review progress, Cross-cutting checks, and Consolidation is ticked.
- Every finding follows the format above and has a unique ID.
- The Summary section at the top is populated.

When all of these are true, output `LOOP DONE` and stop without modifying anything. Do not start a fresh review pass. Do not "polish" findings.

---

## Things not to do

- Don't modify any spec, `SLICES.md`, `VISION.md`, or `ARCHITECTURE.md`. The review is read-only with respect to those.
- Don't fix issues you find. Fixes belong to a different loop.
- Don't take more than one action per iteration. One spec or one cross-cutting check per iteration, no exceptions.
- Don't skip the priority order. Per-spec review comes before cross-cutting checks; both come before consolidation.
- Don't invent findings to pad the report. If a spec is clean, write "No findings." under its heading and tick the box.
- Don't speculate about future slices not yet specced — the review covers what exists.
