## Summary

Slice 006 lands the sync orchestrator end-to-end. Clicking "Sync now" on the Dashboard fires `POST /api/sync`, which acquires an in-memory single-job mutex, walks each connected account's Gmail history (or falls back to a date-range search), classifies each candidate message via Slice 005's pipeline, persists receipt-shaped artifacts to `./invoices/{slug}/{yyyy}/{mm}/...` on disk and to a new `documents` table, and streams progress over `GET /api/sync/events` (SSE). The Inbox view at `/inbox` flips from a live-Gmail read-through (Slice 003) to a DB-backed listing of persisted documents with a Preview link to `GET /api/documents/:id/file`. The dev-seed panel from Slice 004 and the per-row Classify button from Slice 005 are deleted along with their HTTP routes â€” the orchestrator subsumes both.

Net change: **+1138 lines, âˆ’2401 lines** across 27 files (the Slice-004 dev-seed and Slice-005 ad-hoc surfaces account for most of the deletions). The full test suite is **48 files / 455 tests** passing in ~24 s. Two ADRs land: ADR-007 (SSE + ring buffer over WebSocket / long-polling / persisted event log) and ADR-008 (in-memory single-job mutex over a DB-backed `jobs` table or process lockfile).

The spec's Observable result â€” "click Sync, watch progress stream in, find receipts on disk and in the Inbox view" â€” is satisfied at the unit-test layer end-to-end and at the smoke layer for everything verifiable from the agent shell. **Browser-driven AC #1-#10 (the "with Ollama running, real Gmail" path) are deferred to human acceptance** â€” same shape as Slice 005's review. ## ADRs introduced

- `docs/adr/007-sse-with-ring-buffer.md` â€” chose SSE over WebSocket (no need for bidirectional, no extra `ws` dep, browser-native `EventSource`) and over long-polling (per-poll connection cost). The 200-event ring buffer is justified over a persisted `sync_events` table (events duplicate `processed_messages`; "what's happening now" doesn't need archival; Slice 010's Audit view is the persistent surface).
- `docs/adr/008-in-memory-single-job-mutex.md` â€” chose an in-memory `let activeJob` over a DB-backed `jobs` table or filesystem lockfile. Documents the implication: a `docker compose restart` (or `node` exit) drops in-flight job state; partial work survives because `processed_messages` is the idempotency anchor, and the next sync resumes naturally. Acceptable for v1's single-user context.

## Decisions worth flagging

- **`useSyncEvents` does not fetch `/api/sync/status` on mount.** Spec Â§ "Deliverables" â†’ "UI views / components" describes SyncControls counters as "sourced from the SSE stream + GET /api/sync/status snapshot." The hook subscribes to SSE only and relies on the ring-buffer replay (ADR-007) to handle the late-subscriber case. Strictly less code, no parallel REST call, and the ring buffer's 200-event capacity is plenty for the largest realistic in-flight job. If a future spec needs to bootstrap state for an in-flight job whose `sync.start` event has already rolled out of the ring (i.e., > 200 events ago), `/api/sync/status` is the right surface to fetch â€” but this slice's tests confirm it isn't needed at v1 scale.
- **Body-rendering's source-bytes channel.** `classifyMessage` returns `source_bytes?: Map<string, Buffer>` keyed by stable artifact descriptors (`attachment:<filename>`, `body:rendered_html_source`). The map is intentionally non-JSON-serializable so it can't accidentally cross the HTTP boundary; the type lives off the schema. Alternative (passing the bytes back via Gmail re-fetch in the orchestrator) was rejected as wasteful â€” Slice 005's classifier already had the bytes in hand.
- **`refreshSyncState` is best-effort.** After each account's loop, the orchestrator calls `getProfile()` to update `sync_state.last_history_id`. A failure (network blip, rate limit) is silently absorbed in a try/catch â€” the next sync just runs as a range fallback. Hard-failing here would block subsequent accounts on a transient issue; absorbing the failure costs at most one fallback range search next time.
- **Hash-from-bytes dedup over catch-the-UNIQUE.** Documented above under "Fixed during this spec." The spec doesn't prescribe which side handles dedup; the chosen shape is strictly cheaper for the common case (no file write on hit) and equally correct for the uncommon one.
- **`SyncControls.tsx` uses a local `busy` flag + a `sawActive` ref to bridge the clickâ†’SSE-confirm gap.** Without `busy`, the button would re-enable for the moment between a successful POST 202 and the orchestrator's `sync.start` SSE event. The ref-based latch handles the "active ran trueâ†’false" transition cleanly when SSE events arrive across multiple React ticks (production); the test for "re-enables on sync.done" splits the start-then-finish sequence into two `act()` blocks because batched dispatches in a single act() collapse the intermediate `true` state. Production code is correct; the test split reflects production timing. Inline comment in the test names this.
- **`requireKnownAccount` over `requireConnectedAccount` for the documents listing.** Spec Â§ "API endpoints" says "Authorizes implicitly (single user; no auth gate yet)." A `needs_reauth` account's documents are still legitimate to read (they were captured before the token expired). Plan suggested updating `requireConnectedAccount` to call into the new helper; left them parallel because the four-branch logic of the connected variant doesn't share much beyond the initial `findById`. Two clear functions instead of nested helpers.
- **`Document` on the client mirrors the server's `DocumentListRow` (with the JOINed columns).** The route's response shape `{rows: DocumentListRow[]}` already includes the joined `classification` / `confidence` / `subject` / `sender_domain`, so the client type carries them too. If a future surface needs the unjoined `Document` (just the `documents` row, no JOIN), the right shape is splitting `Document` and `DocumentRow` on the client. Not needed yet.
- **`SyncEvent` is a discriminated union over `event` rather than a loose `{event: string, payload: unknown}`.** Cost: the hook's reducer can switch exhaustively and the type-checker catches typos in event names. Benefit pays for itself the moment a new event type lands (Slice 010+ may add `sync.reclassify.start`).
- **Filename sanitizer escapes Windows-reserved names (`CON`, `PRN`, `AUX`, `NUL`, `COM[1-9]`, `LPT[1-9]`).** Step 6's done-log explains: defensive for the export bundle (Slice 011), which lands on accountants' machines that may be Windows. Costs one regex on Linux; pays for itself once.
- **`writeReceiptFile`'s atomic write uses `<final>.tmp` + rename, not a temp dir + rename.** Same parent dir means `rename(2)` is atomic on the same filesystem. If the bind-mount target is on a different filesystem from `/tmp`, a temp-dir approach would silently degrade to a copy. Same-parent-dir keeps the atomicity guarantee.
