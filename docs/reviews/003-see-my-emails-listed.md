# Slice 003: See my emails listed ‚Äî Review

**Spec:** `docs/specs/003-see-my-emails-listed.md`
**Plan:** `docs/plans/003-see-my-emails-listed.md`

## Summary

Docurator now talks to Gmail. The Inbox view ships at `/inbox`, picks an account from the slice-002 picker, and renders the most recent 50 messages' Subject / Sender / Date headers fetched live from `users.messages.list` + `users.messages.get` with `format=metadata` ‚Äî no local persistence. React Router v6 wires `/` (Dashboard) and `/inbox` (Inbox) under a top-of-page Nav with both links and the heading. The Gmail client wrapper (`src/server/gmail/client.ts`) exposes only two read methods (`listMessages`, `getMessage`); the read-only guarantee is enforced at build time by a new `scripts/check-gmail-readonly.ts` substring scanner that fails `npm run build` on any of 24 forbidden Gmail-write substrings ‚Äî captured in **ADR-003**. All nine spec acceptance criteria are covered: AC #7, AC #8, AC #9 verified by the smoke recipe; AC #1‚Äì#6 covered by 14 messages-route tests + 9 Inbox-view tests + 6 AccountPicker tests, with the human-driven half (consent screen, in-browser table render, real Google revoke) recorded as deferred for human acceptance.

The slice ships **49 new tests across 9 new files** (server: 21 in 3 files; client: 24 in 5 files; scripts: 7 in 1 file) and **one ADR**. Three structural decisions are worth a second look: the `as unknown as OAuth2Client` cast in the Gmail wrapper (because `SessionClientLike` is a structural subset of `OAuth2Client` for testability), the FORBIDDEN list's inclusion of `gmail.metadata` and `gmail.labels` as scope tripwires (they're not write methods but they expand read access), and the localStorage-backed picker selection key (the first client-side persistence in the project; the architecture is silent on it).

## What landed

- **DB tables / columns / migrations:** none. This slice is read-only against `accounts` (via `findById`/`updateStatus`), with the only write being the existing `accounts.updateStatus(id, 'needs_reauth')` row-flip path when a session is missing ‚Äî no new tables, no new columns, no new migrations.
- **API endpoints:**
  - `GET /api/accounts/:id/messages?limit=50` ‚Äî returns `{ messages: Array<{ id, thread_id, subject, from, date, internal_date }> }` on 200; 400 on non-integer id or out-of-range limit; 404 on missing row; 409 on `needs_reauth` status (or on connected-but-no-session, with the row flipped to `needs_reauth` as a side effect); 401 `{ error: 'needs_reauth', account_id }` when `withFreshTokens` raises `invalid_grant`; 502 `{ error: 'gmail_error', message }` on any other Gmail error including per-message `getMessage` failures.
- **UI views / components:**
  - `src/client/views/Inbox.tsx` ‚Äî at `/inbox`. Loads accounts on mount, picks initial account (localStorage ‚Üí first connected ‚Üí null), renders `<AccountPicker>` + Subject/Sender/Date table. Loading, empty, needs_reauth, gmail_error, unexpected error states each have their own render branch.
  - `src/client/components/AccountPicker.tsx` ‚Äî reusable `<select>`. Filters by `includeDisconnected`; renders `needs_reauth` rows as `<option disabled>` with a `(needs reauth ‚Äî reconnect on Dashboard)` suffix; renders the empty state when no `connected` accounts exist regardless of `includeDisconnected`.
  - `src/client/components/Nav.tsx` ‚Äî top-of-page header rendering `<h1>Docurator</h1>` + `Dashboard` and `Inbox` `NavLink`s. The `<h1>` moved here from `Dashboard.tsx` so it appears on every route once.
  - `src/client/router.tsx` ‚Äî `<AppRoutes />` declaring two `<Route>`s.
  - `src/client/App.tsx` ‚Äî now `<BrowserRouter><Nav /><AppRoutes /></BrowserRouter>`.
  - `src/client/types.ts` ‚Äî adds `Message` type matching the API response shape verbatim.
- **Files / modules (server):**
  - `src/server/gmail/client.ts` ‚Äî `createGmailClient(accountId)`. Two methods. `setGmailFactoryForTest` / `resetGmailFactoryForTest` for DI.
  - `src/server/api/messages.ts` ‚Äî `registerMessagesRoutes(app, deps?)` with `extractHeader(message, name)` case-insensitive helper, local `isInvalidGrantError(err)` predicate.
  - `src/server/app.ts` ‚Äî registers the messages route between `registerOauthRoutes` and the static fallback.
- **Build-time check:**
  - `scripts/check-gmail-readonly.ts` ‚Äî `scanForForbidden(rootDir, selfPath?) ‚Üí { hits }`. CLI entrypoint when run directly; importable for tests. Walks `.ts/.tsx/.js/.jsx` recursively; skips its own absolute path.
  - `scripts/check-gmail-readonly.test.ts` ‚Äî 7 tests including a positive guard against the real `src/`.
  - `package.json` ‚Äî adds `"check:gmail-readonly": "tsx scripts/check-gmail-readonly.ts"` and chains it as the **first** step of `"build"`.
- **Tooling / config:**
  - `package.json` adds runtime dep `react-router-dom@^6` (resolved to 6.30.3, +3 packages with `react-router` and `@remix-run/router`).
  - `vitest.workspace.ts` ‚Äî server project's `include` extended to `['src/server/**/*.test.ts', 'scripts/**/*.test.ts']`.
  - `tsconfig.json` ‚Äî adds `scripts/**/*` to the include array.
- **Tests:** 9 new files, 49 new tests. Total project: 23 files, 146 tests.
- **Loop artefacts:** `docs/research/003-see-my-emails-listed.md`, `docs/plans/003-see-my-emails-listed.md`, this review.

## ADRs introduced

- `docs/adr/003-build-time-readonly-gmail-check.md` ‚Äî Substring-scanning Node script enforces "no Gmail-write API references in `src/`" at build time. 24 forbidden substrings (matching `architecture.md` ¬ß "Read-only Gmail access" verbatim, plus `gmail.metadata` and `gmail.labels` as scope tripwires). Wired as the first step of `npm run build`. Self-exempts via the script's own absolute path. Rejects: ESLint custom rule, `eslint-plugin-no-restricted-imports`, AST-based `ts-morph` scanner, pre-commit-hook-only, runtime check.

## Code review notes

**Fixed during this spec:**

- *vitest 2.1's generic `spyOn` typings reject `ReturnType<typeof vi.spyOn<T, K>>`.* `gmail/client.test.ts` initially used the inline annotation; `tsc --noEmit` raised TS2344 ("`'withFreshTokens'` does not satisfy the constraint `'never'`"). Worked around with a small `function makeSpy()` factory whose return type is inferred. Reproducible across the codebase if other tests need typed spy variables; flagged for the broader vitest 3 chain upgrade ADR-001 anticipates.
- *Dashboard heading would render twice once `Nav` was added.* Moved the `<h1>Docurator</h1>` out of `Dashboard.tsx`'s render branches and into `Nav.tsx`'s `<header>`. Updated the one Dashboard test that asserted the heading to assert `alice@example.com` instead (the heading contract is now Nav.test.tsx's responsibility).
- *Multi-node `getByText` regex broke.* `Inbox.test.tsx`'s "no accounts connected ‚Äî connect one on the Dashboard" message has the trailing `Dashboard` inside a `<Link>`, so the originally-planned `/no accounts connected.*dashboard/i` couldn't match across two text nodes. Tightened to `/no accounts connected/i` and asserted the link separately via `getByRole('link', { name: /dashboard/i })`.
- *`as unknown as OAuth2Client` cast at Gmail call sites.* `withFreshTokens`'s callback receives a `SessionClientLike`, but `google.gmail({ version: 'v1', auth })` expects an `OAuth2Client`. The runtime value is the real client; only the type narrows. Cast is documented inline with a comment pointing at the research doc.
- *Conditional spread in the SDK call args.* `gmail.users.messages.list({ userId, maxResults, q?, pageToken? })` ‚Äî passing `undefined` for the optional fields would still serialize as keys in the request object, which the SDK then encodes in the query string. Used `...(args.q !== undefined ? { q: args.q } : {})` to keep the request body minimal.
- *`isInvalidGrantError` re-implemented locally in `messages.ts`.* Same predicate as `session.ts` exposes (substring on `Error.message` plus the `err.response?.data?.error === 'invalid_grant'` GoogleAuthError shape). Re-implemented rather than widening session's exported API. If a third caller ever needs the same predicate, extract to `auth/invalid-grant.ts` ‚Äî see follow-up.

**Followups for later (non-blocking, captured for future slices):**

- *Suspected stale `tsx watch` reload.* During the smoke run the user's running dev server returned `500 Internal Server Error` for `GET /api/accounts/1/messages` even though the route's catch-all branch exists and would have returned 502 on any thrown error. The unit tests cover the API contract; the live behavior could not be reproduced because the agent's substitute server failed to bind to the in-use port. Recommend the human stop their `npm run dev` and re-launch before exercising the in-browser smoke. If the 500 reproduces on a fresh launch, it's a real bug to investigate. The most likely cause (best guess) is an old in-memory session entry from a pre-Slice-003 OAuth flow that didn't survive a code reload cleanly ‚Äî the no-session row-flip path covers the post-fix behavior.
- *React Router v6 future-flag warnings printed at every test.* `v7_startTransition` and `v7_relativeSplatPath` are advisory and have no functional effect. Silenced by passing `future={{ v7_startTransition: true, v7_relativeSplatPath: true }}` on `<BrowserRouter>` / `<MemoryRouter>`. Defer to a small polish slice or the v7 migration when it lands.
- *N+1 fetch pattern for the Inbox.* `users.messages.list` returns ids only; we then issue 50 sequential `users.messages.get` calls per Inbox load. The spec calls this out as acceptable for a throwaway slice; Slice 006's sync orchestrator handles concurrency. If this surfaces as user-visible latency before then, batched-with-`Promise.all` `getMessage` calls are a one-line change inside the loop in `messages.ts`.
- *No request-cancel on Inbox picker change.* When the user changes the picker mid-fetch, the in-flight request is abandoned via the `cancelled` flag ‚Äî the response, when it arrives, is just dropped on the floor; the underlying network request and the 51 Gmail API calls keep running. Adding `AbortController` to both `fetch` calls (and threading the signal into the Gmail SDK if it accepts one) would be cleaner. Defer until rate limits actually bite.
- *`StrictMode`-induced double-mount on Inbox.* Inherited from Slice 002's review: in dev mode, `useEffect` fires twice, so `/api/accounts` is fetched twice and (after the picker resolves) `/api/accounts/:id/messages` is fetched twice. The `cancelled` flag handles the React side correctly, but Gmail still bears 102 round-trips per dev-mode mount instead of 51. Production builds don't double-fire. Worth fixing only if the dev-loop friction becomes annoying.
- *Empty `display_name` handling in `AccountPicker`.* The label uses `display_name ?? email`. Slice 008 introduces user-editable `display_name`, at which point users may set empty strings ‚Äî the `??` falls through `null` but not `''`. Defer until the editing UI lands; the right time to harden is when the input shape changes.
- *Future-proof `OAuth2ClientFactory` reuse.* `oauth.ts`, `session.ts`, and now `gmail/client.ts` all have factory-injection seams shaped slightly differently (per-call vs. module-level). Pick one shape across the auth/gmail layer in a small refactor. Not blocking.
- *No `dotenv` dependency for host-side `npm run dev`.* Slice 002's review captured this; `package.json` already uses `tsx watch --env-file=.env`, which works. Ignored.
- *`pollIntervalMs`/`pollTimeoutMs` props on Inbox would be useful.* Not added in this slice because the Inbox doesn't poll ‚Äî it's a one-shot fetch on mount and on picker change. If a future slice adds a "refresh" button that polls, the Slice-002 pattern is the template.

## Decisions worth flagging

- **`gmail.metadata` and `gmail.labels` are in the FORBIDDEN list as scope tripwires.** They aren't write methods but they expand read access beyond `gmail.readonly`. ADR-003 documents the call. **Reasonable disagreement zone:** if a future slice needs read-side metadata access at lower cost (the `gmail.metadata` scope returns metadata for *all* messages without needing per-message fetches), this entry would have to be re-examined. For v1, the strict read-only-via-`gmail.readonly` posture is the architecture's stated property.
- **localStorage key `docurator.lastInboxAccountId` is the project's first client-side persistence.** Stores only the account id (a number). No email content, no tokens, nothing sensitive. Architecture is silent on client storage; this slice picked the obvious convention (`docurator.<feature>` namespace). Future slices that persist UI state should follow this pattern.
- **`as unknown as OAuth2Client` cast at the Gmail call site.** `SessionClientLike` is a structural subset that the production `OAuth2Client` satisfies, but TypeScript can't narrow it through a generic boundary without help. Considered widening `SessionClientLike` to mirror `OAuth2Client` more fully ‚Äî rejected because the test factory shape would have grown to match. The cast is documented inline.
- **Strict 400 on `?limit=` outside `[1, 100]` (spec says default 50, 100 ceiling implicit).** Spec's prose says "the most recent 50 messages" with `limit=50` as the path's default. The 100 ceiling is a defensive choice not in the spec ‚Äî picked to bound the per-request Gmail API cost (51 calls at limit=50, 101 at limit=100). The Inbox UI doesn't expose this knob; adding higher limits later is an off-the-shelf API change. **Reasonable disagreement zone:** the human could prefer a higher ceiling, or no ceiling at all (defer to Gmail's own `maxResults` of 500).
- **Per-row `getMessage` failure fails the whole request with 502.** Spec is silent. Plan stance was "don't return partial results because they'd be confusing in the UI". Alternative: return what we have plus an `errors[]` array. Rejected because Slice 006's sync orchestrator (where partial-failure handling is first-class) is the right place for that; this throwaway-friendly Inbox stays simple. **Reasonable disagreement zone:** 51 sequential calls means a single transient 429 takes the whole page down; a small retry would soften this without crossing the partial-results line.
- **`pickInitialAccount` falls back from a stored-id-not-connected to the first connected, silently.** The spec says "defaults to the most recently used account, falling back to the first `connected` account". Strictly, "falling back" could mean "fall back only when the stored id doesn't exist" ‚Äî implementation also falls back when the stored id exists but is `needs_reauth`. This is the more useful behavior (the user picks the picker, not the dropdown) and matches the spec's intent based on the AC #4 description. Flagged because a strict reading could disagree.
- **Empty state of the Inbox uses inline text + a `<Link>`, not a CTA button.** Visual decision; the architecture doesn't call this out. AccountList's empty state on the Dashboard is similar prose. Tailwind/shadcn polish lands in Slice 016.
- **Dashboard test's heading assertion was dropped, not migrated.** The plan offered "render `<Nav />` alongside `<Dashboard />` in a MemoryRouter, or move the heading assertion into `Nav.test.tsx` and drop it from Dashboard's". Picked the latter ‚Äî each component tests its own contract. The integration-coverage gap (Dashboard rendered together with Nav) is now `router.test.tsx`'s job.

## Deviations from spec or architecture

- **`?limit=` upper bound (100) is not in the spec.** Spec describes `?limit=50` as the path; the 100 ceiling is an implementation defense. No behavioral effect at the spec's default; the Inbox UI doesn't expose the parameter.
- **`createGmailClient` exposes only `listMessages` and `getMessage`, but accepts `q` and `pageToken` on `listMessages` and `format` (any of the four) + optional `metadataHeaders` on `getMessage`.** The spec lists those exact signatures; this implementation matches verbatim ‚Äî no widening, no new methods.
- **Error code strings:** spec says `error: 'needs_reauth'` (401), `error: 'gmail_error'` (502). Implementation matches verbatim. Added: `error: 'invalid_id'` (400 non-integer id), `error: 'invalid_limit'` (400 out-of-range), `error: 'account_not_found'` (404), `error: 'account_not_connected'` (409). The latter four are not in the spec but are necessary discriminators for client error handling; flagged here so the spec author can fold them into the spec's response-shape table if desired.
- **`AccountPicker` empty-state predicate.** Spec describes the picker filtering out non-connected accounts with `includeDisconnected={false}` and rendering disabled rows with `includeDisconnected={true}`. Implementation also renders the empty state when *no* `connected` rows exist regardless of `includeDisconnected` ‚Äî selecting a disabled row is structurally impossible, so showing the dropdown would be misleading. Functionally aligned with the spec's "Inbox can only render messages from a connected account" line.
- **Dashboard heading moved to Nav.** Spec describes `Nav.tsx` as a "top-of-page nav with links to / and /inbox" ‚Äî silent on whether it owns the heading. Moved `<h1>Docurator</h1>` from `Dashboard.tsx` into `Nav.tsx` so the heading appears on every page exactly once. Slice-001's "page contains 'Docurator'" criterion is preserved. No behavioral deviation.
- **`@testing-library/jest-dom` not adopted.** Tests use plain `getAttribute()` / `textContent` assertions rather than `toHaveAttribute` / `toHaveTextContent`. Matches the existing client tests' style (Slice 002 didn't adopt it either). Adding jest-dom would be a one-line `setupFiles` addition; deferred.
- **No deviations from `docs/architecture.md`.** The Gmail client matches ¬ß "Read-only Gmail access" exactly (only `users.messages.list` and `users.messages.get`, with `format=metadata` for `get`). The build-time check matches ¬ß "Security notes ‚Äî CI enforcement" verbatim. The Inbox view matches ¬ß "Components ‚Äî Frontend ‚Äî Inbox" minus the filters and DB-backed read source which are explicitly Slice 006's responsibility.
